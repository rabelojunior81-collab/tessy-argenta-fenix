## PREMISSA

Ferramentas para devs não se vendem com anúncio. Elas se vendem quando um dev mostra para outro dev que funcionam. O conteúdo é o canal de distribuição — mas conteúdo genérico não converte. Conteúdo que resolve uma dor específica, de forma honesta, com demo real, converte.

**Princípio central:** Não seja o produto falando de si mesmo. Seja o fundador resolvendo problemas em público.

---

## PARTE 1 — POSICIONAMENTO DE CONTEÚDO

### 1.1 O Ponto de Vista Único (POV)

Toda estratégia de conteúdo precisa de um ponto de vista claro — algo que você acredita que a maioria discordaria ou não pensou ainda.

**POV da Tessy para conteúdo:**

> "O problema do Vibe Coding não é que as ferramentas são fracas — é que elas te abandonam no momento crítico: quando você precisa de filesystem real, terminal real, e um LLM que sabe o contexto do seu projeto de verdade."

Esse POV é:
- Provocativo (confronta Lovable, Bolt, que são populares)
- Verificável (qualquer dev que tentou escalar um projeto no Lovable sabe que é verdade)
- Posiciona Tessy como a solução naturalmente

### 1.2 Os 3 Pilares de Conteúdo

**Pilar 1 — "Eu resolvi X com Tessy"** (Conversão)
Conteúdo demonstrativo. Mostra o produto em ação resolvendo um problema real.
- Formato: thread Twitter, vídeo curto, post LinkedIn
- Frequência: 2x por semana
- Objetivo: gerar curiosidade e cliques para o produto

**Pilar 2 — "Vibe Coding tem um problema sério"** (Posicionamento)
Conteúdo educativo sobre as limitações do ecossistema atual. Sem atacar concorrentes — apenas nomeando a dor real.
- Formato: thread longa Twitter, artigo LinkedIn, YouTube
- Frequência: 1x por semana
- Objetivo: estabelecer autoridade, criar contexto para a solução

**Pilar 3 — "Como construímos a Tessy"** (Confiança)
Conteúdo de bastidores. Transparência sobre o processo, os erros, as decisões.
- Formato: Twitter informal, LinkedIn mais elaborado
- Frequência: 1x por semana
- Objetivo: construir conexão humana, atrair colaboradores e founders

## PARTE 3 — ESTRATÉGIA POR CANAL

### 3.1 Twitter/X — Posicionamento e Distribuição

**Objetivo:** Canal de aquisição principal para o Segmento A (semi-pro BR)

**Tom:** Direto, sem frescura, técnico mas acessível. Provocativo quando necessário. Em português.

**Tipo de post que performa bem no nicho dev BR:**
- Threads que mostram algo técnico funcionando ("Fiz X com Y, aqui está como")
- Comparativos honestos sem FUD
- Bastidores de construção de produto ("falhei nisso, aprendi aquilo")
- Perguntas que geram debate ("alguém mais acha que Lovable não serve para projetos sérios?")

**Hashtags relevantes:**
`#DevBR` `#VibeCode` `#IA` `#Programacao` `#IndieHackerBR` `#ReactJS` `#TypeScript`

**Frequência mínima:** 3 posts/semana para manter presença. 1 thread semanal para crescimento.

**Regra de ouro:** Responder TODOS os comentários nas primeiras 2h do post. O algoritmo premia engajamento inicial.

### 3.2 LinkedIn — Autoridade e Segmento B/C

**Objetivo:** Construir credibilidade com indie makers e devs mais sêniors. Canal para Segmento B e C.

**Tom:** Mais elaborado que Twitter, mas sem corporativês. Honesto, com dados e raciocínio.

**Formato que funciona no LinkedIn dev:**
- "Aprendi X construindo Y" (lição + backstory)
- Análise de tendências do mercado com opinião própria
- Transparência de negócio (métricas, decisões, erros)
- Celebração de milestones com contexto

**Frequência:** 2-3x por semana. LinkedIn penaliza menos frequência do que Twitter.

**Estratégia de alcance:** Tag estratégica de pessoas relevantes quando genuinamente relacionado ao post. Nunca spam de tags.

### 3.3 YouTube — Conversão Profunda

**Objetivo:** Tutorial completo que converte visitante em usuário. Quem assiste vídeo de 10+ min tem intenção real.

**Tipos de vídeo:**
- Tutorial: "Do zero ao [resultado] com Tessy em X minutos"
- Demo: Feature específica em funcionamento real
- Comparativo: "Tessy vs Claude Code — quando usar cada um"
- Bastidores: "Como construímos a funcionalidade X"

**Frequência:** 1x por mês — qualidade > quantidade no YouTube

**SEO de vídeo:** Títulos com as keywords que o público busca:
- "claude code interface visual"
- "ambiente de desenvolvimento IA browser"
- "terminal no browser typescript"
- "github copilot alternativa gratuita"

### 3.4 Discord — Comunidade e Retenção

**Objetivo:** Canal de retenção, não aquisição. Onde usuários existentes aprofundam engajamento.

**Estrutura sugerida do Discord Tessy:**
```
📣 anúncios
💬 geral
🐛 bugs-e-feedback
💡 ideias-e-features
🏆 showcase (o que você construiu com Tessy)
🔒 founders-circle (acesso restrito)
🤝 parcerias (para biz dev)
🧑‍💻 hackathon-[edição]
```

**Dinâmica de comunidade:**
- Adilson presente ativamente (responder, não só anunciar)
- Promover posts de members no #showcase
- Pedir feedback específico, não genérico: "Como está a experiência de conectar o GitHub?" em vez de "O que você acha?"

### 3.5 GitHub — Credibilidade Técnica

**Objetivo:** Validação técnica para devs que avaliam o produto

**Ações:**
- README impactante com demo GIF ou screenshot da UI
- CONTRIBUTING.md claro — como contribuir com código ou docs
- Issues abertas e marcadas corretamente (good first issue, help wanted)
- Discussions ativas — não só issues
- GitHub Sponsors configurado quando o momento vier

## PARTE 4 — CAMPANHAS ESPECÍFICAS

### 4.1 Campanha "Tessy vs [Concorrente]" (provocativa mas honesta)

**Premissa:** Comparativos honestos geram tráfego enorme. Não é atacar — é mostrar o que cada um faz melhor.

**Formato:**
- Thread de 10+ tweets: "Usei [Lovable/Bolt/Claude Code] para construir X. Aqui está o que funcionou e o que não funcionou. Depois fiz o mesmo no Tessy. Resultado:"
- Post LinkedIn expandido
- YouTube vídeo comparativo (mais longo prazo)

**Cuidados:**
- Sempre reconhecer as qualidades do concorrente
- Nunca inventar ou exagerar limitações
- Focar em casos de uso onde Tessy genuinamente vence

**Temas ideais:**
- Tessy vs Claude Code: "Um tem interface, o outro não. Quando importa?"
- Tessy vs Lovable: "Um vai até o deploy, o outro vai além do sandbox"
- Tessy vs Cursor: "Um é extensão de editor, o outro é browser-native. Qual o impacto?"

### 4.2 Campanha "Quanto Gasta com IA" (transparência)

**Premissa:** Ninguém fala sobre o custo real de IA para dev. Tessy pode ser a voz honesta sobre isso.

**Formato série:**
- "Este mês gastei R$XXX em APIs de IA para construir Tessy. Aqui está o breakdown."
- "Como reduzi meu custo de IA em 60% sem perder qualidade (com roteirização de modelos)"
- "O modelo que você usa importa mais do que você pensa — comparativo de custo × qualidade"

**Por que funciona:** É conteúdo financeiro + técnico. Engaja devs que pagam por LLM e querem otimizar.

### 4.3 Campanha "Build in Public" (bastidores)

**Premissa:** Construir em público é uma das estratégias de crescimento mais eficazes para fundadores solo.

**Formato semanal:**
- Toda sexta no Twitter: "Esta semana em Tessy: [o que foi feito], [o que bloqueou], [o que aprendeu], [o que vem próximo]"
- Não precisa ser polido — autenticidade > perfeição

**Benefícios colaterais:**
- Atrai colaboradores (ver Documento 03)
- Cria documentação orgânica do processo
- Humaniza o produto

### 4.4 Campanha "Founders Circle — Convocação" (recrutamento)

**Formato:**
- Post anunciando que está selecionando os primeiros 20 membros do Founders Circle
- Critério claro: quem já está usando ou quer usar ativamente
- CTA: "DM ou formulário para expressar interesse"
- Prazo artificial: "Apenas as primeiras 20 vagas"

**Timing:** Após 4 semanas de conteúdo — quando houver engajamento suficiente para gerar interesse real.
