# Phase 4.1: Tessy AI - Context

**Gathered:** 2026-04-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Entregar o Chat interface com multi-provedores de IA (streaming em tempo real), Janela de Contexto (sliding window baseada em tokens) e execução de ferramentas limitadas à sandbox do workspace. Diferente de um webapp de chat isolado, a Tessy é uma IDE (Copilot) em que a IA gera edições em código reais manipuladas pelo Monaco Editor, consolidando a inteligência artificial ao ecossistema local.

</domain>

<decisions>
## Implementation Decisions

### 1. Runtime da Orquestração
- **TypeScript Nativo no Backend Próprio da Tessy:** A lógica do Gateway (streaming, provedores agnósticos) será implementada dentro do próprio backend da Tessy (`tessy-antigravity-rabelus-lab/server/` + `services/aiProviders.ts`). **Não** haverá microserviços em Python nem dependência runtime do `inception-v2`. As requisições cruas e os buffers assíncronos que dão a fluidez do Open WebUI serão desenhados via Fetch API/Streams em JS, espelhando o **padrão arquitetural** de Provider Polymorphism que o Inception v2 usa em seus `packages/providers/` — como referência de design, não como dependência de runtime.

### 2. Arquitetura RAG (Context Window)
- **Motor Dual (Local + API):** A injeção de contexto usará duas frentes nativas no TypeScript:
  1. Uso da API Gemini Embeddings 2 para processamento robusto em nuvem.
  2. Chamadas HTTP diretas para um host local do **Ollama** (rodando modelos como `nomic-embed-text` ou `gemma`), garantindo uma solução 100% local-first sem depender da biblioteca `SentenceTransformers` do Python. O usuário poderá escolher um ou ambos.

### 3. Injeção Dinâmica de Tools (Workflows)
- **Schemas Zod + Interceptador Funcional:** As ferramentas (Tools) do Agent/Copilot serão funções fortemente tipadas com `zod`. O motor da Tessy (Broker) intercepta o schema da Tool e hidrata variáveis seguras e sensíveis invisivelmente antes da execução, sem vazar credenciais ou variáveis de ambiente para a interface da UI.

### 4. Ideologia de Artefatos e UI (Canvas)
- **Artefatos como Código (IDE-First):** A saída da IA não gera apenas "widgets de chat", mas propõe edições em arquivos reais (Criação, Sobrescrita, Modificação de código). A visão central é a mesma do VS Code/Antigravity. O código reside no File Explorer e no Monaco Editor.
- **IFrame Sandbox Seguro (Apenas Preview):** A arquitetura genial de `iframe` sandboxed restrito com ajuste de `postMessage` capturada da engenharia reversa do Open WebUI será adotada estritamente como uma **Janela de Preview Seguro (Renderização)** visual de componentes. O estado do artefato pertence ao File System/Editor; a UI apenas espelha esse estado para visualização do usuário quando aplicável (ex: renderizar um botão React gerado).

### 5. O Gateway HTTP Puro e o Auto Doc Engine
- **Polimorfismo de API REST (Sem Vercel SDK):** A dependência sugerida anteriormente de `@ai-sdk` (Vercel) foi sumariamente **rejeitada**. Em vez disso, a Tessy construirá classes de Provedores baseadas em "Estilos" (ex: `OpenAIStyleProvider`, `AnthropicStyleProvider`, `GeminiStyleProvider`, `OllamaStyleProvider`), manipulando o `fetch` manualmente para manter o controle absoluto de payloads e streams (SSE).
- **Auto Doc Engine Roteado:** O "Auto Doc Engine" não carregará apenas documentações de bibliotecas Node; ele deve ser expandido e sincronizado com as especificações puras de API REST para esses "Estilos" de provedores (ex: rotas do OpenRouter, Opencode GO/Zen). Isso permitirá que a Tessy e as IAs se auto-curem e gerem as chamadas perfeitamente para qualquer ecossistema, cumprindo a visão da Incepção.

</decisions>

<specifics>
## Specific Ideas

- O Ollama expõe embeddings diretamente via HTTP (`localhost:11434/api/embeddings`), tornando o RAG local poderoso perfeitamente viável em TypeScript.
- A abstração dos provedores (Gateway) segue o polimorfismo do Open WebUI, abolindo SDKs que ocultem a comunicação com os modelos.

</specifics>
