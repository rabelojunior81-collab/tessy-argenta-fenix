# Coding Conventions

**Analysis Date:** 2026-04-20

## Naming Patterns

**Files:**
- Treat conventions as layer-local, not workspace-global: product code in `tessy-antigravity-rabelus-lab/` uses different naming from platform code in `inception-v2/` and methodology code in `inception-tui/`.
- Product layer React components use PascalCase `*.tsx` in `tessy-antigravity-rabelus-lab/components/` and `tessy-antigravity-rabelus-lab/contexts/` (`components/layout/MainLayout.tsx`, `components/viewers/ProjectsViewer.tsx`, `contexts/WorkspaceContext.tsx`).
- Product hooks use the React `useX` prefix in `tessy-antigravity-rabelus-lab/hooks/` (`hooks/useProjects.ts`, `hooks/useViewerRouter.tsx`).
- Product services use camelCase service names with semantic suffixes such as `Service`, `Client`, `Adapter`, or provider folders (`services/fileSystemService.ts`, `services/brokerClient.ts`, `services/fsaAdapter.ts`, `services/gemini/contextManager.ts`).
- Platform modules in `inception-v2/packages/*/src/` prefer kebab-case or lower-case filenames for non-UI code (`packages/agent/src/message-adapter.ts`, `packages/tools/filesystem/src/path-guard.ts`, `packages/protocol/src/mission-protocol.ts`).
- Platform TSX files use PascalCase when they are Ink UI components (`inception-v2/packages/channels/cli/src/components/App.tsx`, `StatusBar.tsx`, `ApprovalPrompt.tsx`).
- Platform tests are colocated as `*.test.ts` next to implementation (`inception-v2/packages/core/src/runtime.test.ts`, `packages/config/src/loader.test.ts`).
- Methodology layer code in `inception-tui/src/` prefers short lower-case names and `index.ts` barrels (`src/commands/mission.ts`, `src/generators/index.ts`, `src/onboarding/index.ts`).
- Operational workflow docs at root use kebab-case Markdown names in `.codex/get-shit-done/workflows/` (`map-codebase.md`, `verify-phase.md`, `validate-phase.md`).
- Methodology/governance artifacts generated by `inception-tui/src/commands/mission.ts` use uppercase underscore Markdown filenames such as `MISSION_BRIEFING.md`, `TASK_MANIFEST.md`, `COMMUNICATION_PROTOCOL.md`, and `REPORT_TEMPLATE.md`.

**Functions:**
- Use camelCase for functions and methods across all three repos (`loadWorkspace`, `assistantToOutbound`, `missionNew`, `resolveConfig`).
- Prefix React hooks with `use` and keep them in hook modules (`tessy-antigravity-rabelus-lab/hooks/useProjects.ts`).
- Name event handlers and UI actions with verb-first helpers such as `handleViewerResize`, `handleInput`, `switchProject`, and `approveWorkspaceAction` in `components/layout/MainLayout.tsx`, `packages/channels/cli/src/components/App.tsx`, and `contexts/WorkspaceContext.tsx`.
- CLI command exports in `inception-tui/src/commands/mission.ts` are imperative and task-shaped (`missionNew`, `missionList`, `missionArchive`).
- Inception v2 store methods in `inception-v2/packages/memory/src/db/queries.ts` deliberately use snake_case method names (`insert_one`, `get_by_id`, `count_at_depth`) to mirror SQL semantics. Preserve that style inside the database access layer only.

**Variables:**
- Use camelCase for locals, state, and parameters (`currentProjectId`, `workspaceRegistry`, `effectiveToken`).
- Use UPPER_SNAKE_CASE for process-level constants and config sentinels (`PORT`, `HOST`, `SESSION_TTL_MS` in `tessy-antigravity-rabelus-lab/server/index.hono.ts`; `CONFIG_FILENAME`, `AGENT_DIR` in `inception-tui/src/utils/config.ts`).
- Inception v2 class internals may use underscore-prefixed private fields (`_state`, `_startedAt`, `_stats` in `inception-v2/packages/core/src/runtime.ts`).
- Do not introduce underscore prefixes in Tessy React state or service variables unless a file already follows that pattern.

**Types:**
- Use PascalCase for props, context, schema, and local interfaces (`MainLayoutProps`, `WorkspaceState`, `ResolvedConfig`, `StatusBarProps`).
- Local interfaces and type aliases do not use the `I` prefix, but published platform contracts imported from `@rabeluslab/inception-types` sometimes do (`IRuntime` in `inception-v2/packages/core/src/runtime.ts`). Match the surrounding package rather than renaming imported contracts.
- Keep enum names and exported type names PascalCase when consuming `@rabeluslab/inception-types` (`RuntimeState`, `AgentNature`, `MessageRole`).

## Code Style

**Formatting:**
- `tessy-antigravity-rabelus-lab/` is Biome-governed by `tessy-antigravity-rabelus-lab/biome.json`: 2 spaces, 100-character line width, single quotes, double quotes in JSX, semicolons required, trailing commas `es5`, import organization enabled.
- `inception-v2/` is Prettier-governed by `inception-v2/.prettierrc`: 2 spaces, 100-character width, single quotes, semicolons required, trailing commas `es5`, `endOfLine: lf`.
- `inception-tui/` has no formatter config file. Its current source style is semicolon-free, compact, and aligned with manual CLI formatting (`inception-tui/src/cli.ts`, `src/commands/mission.ts`, `src/utils/config.ts`). Preserve that repo-local style when editing there.
- Keep `.js` import specifiers in TypeScript ESM code where the repo already compiles that way (`inception-v2/packages/core/src/runtime.ts`, `inception-tui/src/cli.ts`).

**Linting:**
- `tessy-antigravity-rabelus-lab/` runs `biome check .` from `package.json`. `noExplicitAny` is a warning, not an error, via `biome.json`.
- `inception-v2/` runs `eslint . --ext .ts,.tsx` from `package.json`. The root config in `inception-v2/.eslintrc.cjs` enforces TypeScript strictness, `import/order`, `prefer-const`, `no-var`, and warns on `console.log`.
- Root ESLint in `inception-v2/.eslintrc.cjs` ignores `**/*.test.ts`; package test files are typechecked and styled by local package configs instead of the root lint pass.
- `inception-tui/` has no linter config detected. The only static guard there is `strict: true` in `inception-tui/tsconfig.json`.

## Import Organization

**Order:**
1. Built-in or external packages first (`node:*`, `react`, `ink`, `chalk`, SDK packages).
2. Internal workspace packages next in platform code (`@rabeluslab/inception-types`, sibling workspace packages).
3. Relative imports last (`./runtime.js`, `../utils/config.js`, `../../contexts/WorkspaceContext`).
4. Separate type-only imports with `import type` when the repo already does so (`inception-v2/packages/config/src/loader.ts`, `tessy-antigravity-rabelus-lab/contexts/WorkspaceContext.tsx`).

**Grouping:**
- `inception-v2/.eslintrc.cjs` explicitly requires blank lines between import groups and alphabetical ordering within groups.
- Tessy and inception-tui follow the same broad grouping pattern even where it is not machine-enforced (`components/layout/MainLayout.tsx`, `inception-tui/src/cli.ts`).
- Prefer one import per concern instead of mixed type/value destructuring when files are already written that way.

**Path Aliases:**
- Tessy defines `@` to the repo root in `tessy-antigravity-rabelus-lab/vite.config.ts`, plus a `buffer` alias. Most current app code still uses relative imports.
- No comparable root path alias is configured in `inception-v2/` or `inception-tui/`; prefer package names or relative imports there.

## Error Handling

**Patterns:**
- Tessy product code catches errors near UI or browser/service boundaries, logs context with `console.error` or `console.warn`, and returns nullable or boolean status for expected failures (`contexts/WorkspaceContext.tsx`, `services/fileSystemService.ts`).
- Inception v2 platform code throws typed errors for invalid runtime states (`packages/core/src/runtime.ts`) and uses `Result` return values for expected config failures (`packages/config/src/loader.ts`).
- Inception-tui CLI helpers return `null` for unreadable config (`src/utils/config.ts`) and call `process.exit()` for user-cancel or fatal command states (`src/cli.ts`, `src/commands/mission.ts`).
- Use early returns and guard clauses before the main branch of work. This is common in all three repos.

**Error Types:**
- Throw when invariants are broken or a caller misuses an API (`useWorkspace` without provider in `contexts/WorkspaceContext.tsx`, runtime state transitions in `packages/core/src/runtime.ts`).
- Return `null`, `false`, or a `Result` object for recoverable operational failures such as missing config, canceled file pickers, or unavailable broker state.
- Keep user-facing messages localized to the repo context: Tessy and inception-tui frequently use PT-BR strings, while Inception v2 core/platform code is mostly English.

## Logging

**Framework:**
- No shared logger abstraction is enforced across the workspace.
- Tessy uses `console.log`, `console.warn`, and `console.error` directly in product and server code (`services/fileSystemService.ts`, `contexts/WorkspaceContext.tsx`, `server/index.hono.ts`).
- Inception v2 allows `console.warn`, `console.error`, and `console.info` by lint policy, but discourages broad `console.log`.
- Inception-tui primarily communicates through `@clack/prompts`, `chalk`, and `console.log` (`src/cli.ts`, `src/commands/mission.ts`).

**Patterns:**
- Include a subsystem prefix when logging operational failures (`[WorkspaceContext]` in `tessy-antigravity-rabelus-lab/contexts/WorkspaceContext.tsx`, `[Broker/Hono]` in `server/index.hono.ts`).
- Log at integration boundaries, not inside trivial helpers.
- When logging suppression-worthy behavior, explain it in a nearby comment instead of leaving an unexplained ignore.

## Comments

**When to Comment:**
- Comment the why, lifecycle status, or governance constraint, not the syntax. Tessy heavily documents migration state, gates, and risk notes in `server/index.hono.ts`, `playwright.config.ts`, and `src/test/setup.ts`.
- Inception v2 uses short JSDoc blocks for exported functions, public classes, and conversion helpers (`packages/core/src/runtime.ts`, `packages/agent/src/message-adapter.ts`, `packages/config/src/loader.ts`).
- Inception-tui uses banner comments to segment CLI flow and command groups (`src/cli.ts`, `src/commands/mission.ts`).
- Inline ignore comments must include the exact reason. This is standard in Tessy (`biome-ignore ...: resize handle`, `biome-ignore ...: modal backdrop`) and appears occasionally in Inception v2 (`eslint-disable-next-line @typescript-eslint/no-require-imports`).

**JSDoc/TSDoc:**
- Use docblocks for exported platform APIs and non-obvious helpers in `inception-v2/`.
- Tessy uses a mix of JSDoc blocks and operational header comments; match the file’s existing voice instead of forcing formal TSDoc everywhere.
- Inception-tui favors section banners over formal API docs. Keep that style in the methodology layer.

## Function Design

**Size:**
- Inception v2 favors small, typed functions and focused class methods (`packages/agent/src/message-adapter.ts`, `packages/config/src/loader.ts`).
- Tessy allows large orchestrator files in contexts and UI containers (`contexts/WorkspaceContext.tsx`, `components/layout/MainLayout.tsx`), but still extracts reusable service functions into `services/`.
- Inception-tui command modules can be long, but they split work into small helpers such as `cancelGuard`, `slugify`, and `getMonthTag` in `src/commands/mission.ts`.

**Parameters:**
- Prefer explicit typed parameters and default values over loose option bags when arity is small (`missionNew(dir = process.cwd())`, `openDirectory(mode = 'read')`).
- Use options objects for optional behavior in platform APIs (`assistantToOutbound(..., options = {})` in `packages/agent/src/message-adapter.ts`).
- Use narrow literal unions when the domain is fixed (`kind: 'file' | 'directory'`, runtime state enums, CLI options).

**Return Values:**
- Return booleans or nullable values from UI/service boundaries when the caller needs a simple success signal (`selectDirectory`, `cloneFromGitHub`, `openDirectory`).
- Return typed domain objects from platform APIs and avoid implicit `any`.
- Use explicit `Promise<void>` and `Promise<boolean>` signatures for async command and state-transition methods.

## Module Design

**Exports:**
- Tessy defaults export React components (`components/layout/MainLayout.tsx`) and uses named exports for hooks, services, and contexts (`hooks/useProjects.ts`, `services/fileSystemService.ts`).
- Inception v2 strongly prefers named exports within source files and re-exports public API from `src/index.ts` per package (`packages/core/src/index.ts`, `packages/config/src/index.ts`).
- Inception-tui uses named exports in helpers and command modules, with `index.ts` barrels in generator/onboarding folders.

**Barrel Files:**
- Use `index.ts` as the public package surface in `inception-v2/packages/*/src/index.ts`.
- Use local `index.ts` barrels in `inception-tui/src/onboarding/` and `src/generators/` when the folder already behaves as a package.
- Tessy does not rely on pervasive barrel files. Prefer direct imports between adjacent component, hook, and service files unless a directory already exposes a stable entry point.
- Across the exossistema, match the conventions of the active layer instead of trying to normalize product, platform, methodology, and operational code into one style.
