## Resumo Executivo

A execução da Fase 4.1 foi um **desastre arquitetural**. Em vez de evoluir o sistema existente, criou-se um paralelo disfuncional, desconectado e com modelos deprecados. O resultado final foi revertido por completo: os novos componentes não permaneceram no estado atual do repositório e foram preservados apenas na pasta de arqueologia.

## Linha do Tempo de Recuperação

1. `b65e951` — rollback do root, removendo os artefatos de planejamento e documentação da Fase 4.1.
2. `d7a839b` — rollback do módulo `tessy-antigravity-rabelus-lab`, removendo a implementação executada.
3. `f670542` — sincronização do ponteiro do superproject para apontar o módulo revertido.
4. `scripts/sync/install-superproject-sync.ps1` + `.git/hooks/post-commit` — correção do shim que escapava `"$@"` como texto literal e quebrava o hook no PowerShell.

## Causa Raiz Operacional

A primeira tentativa de undo atingiu apenas o root. A execução real estava no módulo `tessy-antigravity-rabelus-lab`, então o rollback completo exigiu três camadas:

- remover o plano e a documentação do root;
- reverter os commits de implementação do módulo;
- sincronizar o gitlink do superproject.

Esse erro de escopo foi o principal aprendizado operacional deste evento.

## Falhas Identificadas pelo Operador

### 1. DESRESPEITO TOTAL À ARQUITETURA EXISTENTE

**O que existia (funcional):**
- `services/authProviders.ts` — Sistema completo de autenticação com abas (Gemini, OpenAI, Z.ai, GitHub, Firecrawl)
- Central de Autenticação (`AuthPanel` / `Central de Autenticação`) — Modal com tabs, validação de tokens, links para obter chaves
- Sistema de persistência via IndexedDB (`tessy_auth`)
- Validadores por provider (`validator: (token) => ...`)

**O que foi criado (inútil):**
- `components/ai/ProviderSelector.tsx` — Dropdown hardcoded com modelos DEPRECADOS
- Seção "Provedores e chaves" no `ControllersModal` — Inputs básicos sem integração com o sistema real de auth
- Sem conexão com `services/authProviders.ts`
- Sem conexão com `getToken()` / `setToken()` do authProviders

**Erro crítico:** O executor NÃO analisou `services/authProviders.ts` antes de implementar. Se tivesse analisado, teria visto que o sistema já existia e deveria ser **extendido**, não recriado.

### 2. MODELOS DEPRECADOS E HARDCODED

**Modelos no ProviderSelector (DEPRECADOS):**
```typescript
const PROVIDERS = [
  { id: 'openai', label: 'OpenAI', model: 'GPT-4o' },
  { id: 'anthropic', label: 'Anthropic', model: 'Claude Sonnet' },
  { id: 'gemini', label: 'Google Gemini', model: 'Gemini 2.0 Flash' },
  { id: 'ollama', label: 'Ollama (Local)', model: 'Qwen 2.5 Coder' },
];
```

**Problemas:**
- Modelos são STRINGS HARDCODED, não dinâmicos
- Não há hook de verificação de modelos disponíveis
- Não há atualização automática de lista de modelos
- Não há suporte a modelos novos sem recompilar
- Ollama mostra apenas "Qwen 2.5 Coder" — deveria listar todos os modelos locais disponíveis via `/api/tags`

**O que deveria existir:**
- Hook `useAvailableModels()` que consulta a API de cada provider
- Para OpenAI: `GET /v1/models`
- Para Anthropic: lista estática ou API de modelos
- Para Gemini: lista de modelos da Google
- Para Ollama: `GET /api/tags` para listar modelos locais
- Cache com stale-while-revalidate

### 3. OLLAMA MAL IMPLEMENTADO

**O que foi feito:**
- Input hardcoded `http://localhost:11434`
- Modelo hardcoded "Qwen 2.5 Coder"
- Sem integração real com Ollama

**O que deveria existir:**
- Configuração de URL do servidor Ollama (padrão: localhost:11434)
- Botão "Verificar conexão" que faz `GET /api/tags`
- Lista de modelos disponíveis retornada pelo Ollama
- Botão "Baixar modelo" que usa `POST /api/pull`
- Status de cada modelo (baixado, baixando, não disponível)

### 4. MODAL DE PARÂMETROS DESRESPEITADO

**O que existe:**
- `ControllersModal.tsx` com abas "Config" e "Atalhos"
- Já tem fatores configuráveis (Tom, Modelo, Formato, Grounding, Nível de Detalhe, etc.)

**O que deveria ter sido feito:**
- Adicionar o seletor de Provider nos **Parâmetros existentes**
- Substituir ou complementar o fator "model" existente
- Manter a coerência visual e funcional
- NÃO criar um modal separado de "Provedores e chaves"

**O que foi feito (errado):**
- Criou uma seção "Provedores e chaves" no final do modal
- Inputs básicos que não se integram ao sistema real
- Duplicação de funcionalidade (authProviders já fazia isso)

### 5. ROTEAMENTO NÃO IMPLEMENTADO

**Pergunta do operador:** "COMO ESTÁ O ROTEAMENTO ALI? O QUE ESTÁ MANDANDO?"

**Resposta:** O roteamento **NÃO EXISTE** funcionalmente. O `ChatContext.tsx` tem código de streaming, mas:
- Não está integrado ao `sendMessage` real
- O `ProviderRegistry` é instanciado mas não conectado às chaves reais
- O `applyFactorsAndGenerate` do Gemini ainda é o caminho principal
- O novo sistema de providers é **código morto** — existe mas não é executado

### 6. FALHA DE PESQUISA E ANÁLISE

**O que o executor deveria ter feito:**
1. Ler `services/authProviders.ts` COMPLETAMENTE antes de tocar em qualquer auth
2. Entender como `getToken()` e `setToken()` funcionam
3. Entender a estrutura da `Central de Autenticação`
4. Ver que a aba "Gemini" já existia e funcionava
5. Extender o padrão existente para Anthropic e Ollama

**O que o executor fez:**
- Ignorou completamente o sistema existente
- Criou paralelos disfuncionais
- Não leu o código existente antes de implementar

## Lições Aprendidas

1. **NUNCA implementar auth/token/key management sem ler o sistema existente primeiro**
2. **NUNCA hardcode modelos de IA — sempre buscar dinamicamente**
3. **SEMPRE integrar com sistemas existentes em vez de criar paralelos**
4. **VERIFICAR se o código novo é realmente executado (não só existe)**
5. **TESTAR a integração, não só a compilação**
